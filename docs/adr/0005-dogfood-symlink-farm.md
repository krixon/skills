# 0005 — Dogfood the plugin via a faithful symlink farm; keep repo-local skills out of the ship set

## Status

Accepted.

## Context

The repo *is* the `krixon/skills` plugin and dogfoods itself by symlinking its whole skills tree as project skills: `.claude/skills -> ../skills`. That whole-dir symlink makes `.claude/skills/` identical to `skills/`, so every skill's progressive-disclosure link (`../GITHUB.md`, `../HANDOVER.md`, `../contracts/…`) resolves and the live edit loop works with zero ceremony.

The cost: the ship set and the dev set are the same directory. `skills/` is what a marketplace install discovers, and the whole-dir symlink forces every dev-loadable skill to live there. `release` is repo-specific — it bumps *this* `plugin.json` and tags `v*` on *this* `main`; it is meaningless to a consumer using these skills on their own project — yet it ships, and `land` hardcodes a `release` hop, leaking a repo-only skill into the distributable.

Two constraints bound the fix. `plugin.json`'s `skills` field is additive-only — the default `skills/` is always scanned and nothing excludes a subdirectory — so a manifest allowlist can't subset what ships; `release` must *physically* leave `skills/`. And the `../shared.md` links resolve **logically**, against the `.claude/skills/` path (verified: a skill's `../GRILL-METHOD.md` reads as `.claude/skills/GRILL-METHOD.md`), so a partial farm that symlinks only a hand-picked subset of skill dirs breaks every shared-doc link.

## Decision

Separate the dogfood-load set from the ship set, and load the distributable skills through a faithful per-skill symlink farm instead of one whole-dir symlink.

- `skills/` holds exactly the **distributable** skills — `release` is removed from it.
- `.claude/skills/` becomes a real directory containing one symlink per `skills/*` entry (every skill dir **and** every shared doc and `contracts/`, so `../GITHUB.md` still resolves) plus `release/` as a real, tracked directory living directly there. `.claude/` is already tracked, so `release` stays versioned.
- The farm is generated by `bin/relink-dev-skills.sh`; adding a skill reruns it. This is a step in the CLAUDE.md add-a-skill rule (which also stops describing `.claude/skills` as a single symlink). It is *not* added to `write-skill`, which ships: naming this repo's dogfood script in a distributable skill would re-introduce the leak this ADR removes. `write-skill` defers to the project's add-a-skill rule, where the step belongs.
- `land` no longer names `release`. Its post-land hop generalizes to *offer the project's release process if the project's CLAUDE.md documents one* — discovery via the in-context CLAUDE.md, documented-only, never hunting or inferring a release command. This repo's CLAUDE.md documents `release` under a `## Releases` section, closing the loop. The material-change gate (range since last `v*` tag, Conventional-Commit material types) leaves `land`; the release process owns the "anything worth shipping?" judgment.
- `release` and its hop are scrubbed from the distributable surface: the README skills list / I-want-to table, `docs/skills-reference.md`, `HANDOVER.md`'s interactive-only set, and `GITHUB.md`'s tag-push reference.

## Considered Options

- **Consume the published plugin the standard way.** Rejected: marketplace installs are copied to a cache, not loaded live, so working-tree edits wouldn't load — it kills the dogfood loop, which is the symlink's entire purpose.
- **`--plugin-dir .` for the distributable + `release` as a lone project skill.** Rejected: it double-wires hooks (settings.json *and* the plugin's `hooks.json`) or forces hooks out of settings.json into plugin-load — losing hot-reload and, worse, making the guard-rail hooks (`worktree-only-edits`, `security-boundary`) depend on remembering a launch flag. Forget the flag → guard rails off.
- **Symlink `release` into `skills/` and gitignore it.** Rejected: breaks the auditable invariant *everything in `skills/` ships*, and risks a silent ship if the ignore is dropped or bypassed.
- **`.gitattributes export-ignore` on `skills/release/`, keep the whole-dir symlink.** Rejected as primary: marketplace packaging copies/clones the tree rather than `git archive`-ing it, so `export-ignore` likely doesn't apply and `release` would ship silently. Cheaper than the farm *if* verified, but unverified and silent-fail.

## Consequences

The farm costs a regen step on every skill add (`bin/relink-dev-skills.sh`, hung on the add-a-skill rule) and replaces a clean one-line symlink with a ~30-entry generated directory — the trade for an honest, auditable `skills/` == ship-set and a `release` that loads live without shipping.

Moving `release` one level deeper (`skills/release/` → `.claude/skills/release/`) rewrites its links that reach **repo-root** docs: `../../ISOLATION.md` becomes `../../../ISOLATION.md`. Its **skills-level** links (`../GITHUB.md`, `../HANDOVER.md`) are unchanged, because the farm mirrors those docs beside it at the same relative depth. So one repo-local skill carries a `../`-deeper link convention than its distributable siblings — a contained asymmetry, the price of living outside `skills/`.

`land` becomes genuinely project-agnostic, which realigns it with the repo's own stated rule that skills impose no methodology and discover through the project's CLAUDE.md — the `release` coupling was a latent violation of that, not just a packaging wart. Risk: the change must land coherently — a half-migrated farm (some shared docs unmirrored) breaks progressive-disclosure links, so the acceptance check is that every `skills/*` entry is mirrored and no distributable artifact names `release`.
